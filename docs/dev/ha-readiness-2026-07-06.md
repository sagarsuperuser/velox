# HA readiness — SPOF register and the multi-replica plan (2026-07-06)

Single-instance is the supported v1 deployment shape: one `velox-api` process, one Postgres, no load balancer, "~minutes of downtime per deploy/restart is acceptable" ([docs/self-host.md](../self-host.md), Operational posture, lines 55-66). Money paths are crash-safe by construction — invoice generation is idempotency-index exactly-once (internal/subscription/postgres.go:1089-1094), dunning transitions are exactly-once CAS (#325), outbound email/webhooks are durable Postgres outboxes with lease claims, and every scheduler sweep re-derives eligibility from durable state — so process death costs latency, not correctness. What single-instance does NOT give you is redundancy, and what N=2 does not yet give you is safety on a handful of named surfaces. This document is the pre-work for the multi-replica build: every single point of failure, every verified hazard of running two API replicas today, and the ordered work list — so that when a design-partner pilot approaches production cutover (the named trigger, docs/self-host.md:67-69), the build is execution, not discovery. Every claim below carries the file:line it was verified against; surfaces that are already safe say so explicitly.

## SPOF register

Legend: **Lost** = irrecoverable without external action. **Delayed** = self-heals on restart with zero operator work.

### Tier: ingest edge

| Component | Failure | Blast radius | Lost vs delayed | Operator mitigation today |
|---|---|---|---|---|
| Single API process — the only front door for usage. All three ingest doors (`POST /v1/usage-events`, `/batch`, `/v1/integrations/litellm/spend`) are synchronous handlers writing straight to Postgres in-request; no Velox-side queue/WAL/buffer exists (usage/handler.go:146-167; internal/integrations/litellm/handler.go:104-157; internal/usage/postgres.go:110-117) | Process down = connection refused at the edge; Postgres down with process up = failure per-door (see Downtime = money) | Every usage event fired during the window is refused. Nothing already accepted is ever lost — writes are synchronous, so 2xx means committed (usage/postgres.go:110-117) | **Split**: delayed for clients that retry non-2xx; **permanently lost** for clients that don't — including LiteLLM at its defaults (see Downtime = money). ## What breaks at N=2 today

Verified hazards only, ordered by severity. Everything not listed here was checked and is in the verified-safe list below.

### Hazards

**1. [SHIPPED 2026-07-06 — per-invoice CAS claim, migration 0141] The auto-charge sweep was the one lock-gated money path with no row claim — a Postgres-failover dual-leader window could double-charge one invoice.** Fix landed as `ClaimAutoCharge` (dunning-shaped CAS re-asserting the full eligibility predicate, 5m lease, DB-side now(), updated_at untouched for idempotency-key stability) at the top of the `processAutoCharge` loop, with release-on-skip for the two provably-pre-Stripe paths. Deliberately NOT claim-in-list: a batch-stamped lease cannot cover a serial 50-invoice loop, and the list methods are shared with dunning enrollment, which a claiming list would starve (adversarial review). Pinned by: real-Postgres claim collision, dual-leader charges-once, predicate-recheck matrix, updated_at-stability, claim-blind lists, release-on-skip tests. Original finding: A failover releases session advisory locks while the old leader's tick keeps running: the lock pins one conn (advisory_lock.go:44-47, 62-79), tick work runs on other pool conns that reconnect to the new primary, the scheduler never rechecks lock liveness mid-tick (scheduler.go:271-282), and `Release()` swallows errors (advisory_lock.go:90). During the overlap, every duplicated step collapses via SKIP LOCKED / CAS / idempotency-index — except the auto-charge sweep: `ListAutoChargePending` is a plain SELECT with no claim (invoice/postgres.go:1813-1827), and its only dedupe is the Stripe idempotency key derived from `inv.UpdatedAt` (payment/stripe.go:451-484). The divergent-key path: leader A's charge outcome persists as failed/unknown via `UpdatePayment` (bumps `updated_at`, status stays finalized, invoice/postgres.go:703-709); leader B's refresh (engine.go:5339-5344) then mints a NEW key — `chargeInvoice` checks only Status and amount_due, never payment_status (stripe.go:409-426). Double charge fires when A's outcome was UNKNOWN but the PI actually succeeded. Rare (failover coinciding with a tick), customer-visible, refundable, never silent (both PIs emit webhooks). Fix: give the sweep the dunning-shaped CAS claim (a bare `FOR UPDATE SKIP LOCKED` is insufficient — the list tx ends before the charge loop).

**2. Rolling-deploy binary skew permanently DLQs money emails of a new `email_type`.** `email_outbox.email_type` deliberately has no CHECK constraint (outbox_sender.go:40-43, exercised by ADR-082's `TypeCreditNote`), the old binary's dispatcher returns `ErrPayloadDecode` for unknown types (dispatcher.go:189-190), which is classified permanent (outbox.go:81-87) and DLQ'd with zero retries (outbox.go:338-343). The per-tick dispatcher lock is won arbitrarily by either binary (dispatcher.go:118-127), so new-type rows claimed by the OLD binary in the overlap window are dead — not delayed. A per-row `slog.Error` fires (outbox.go:344-351) but `FailedCount` has zero callers and no redrive tooling exists (verified by grep); recovery is hand-written SQL. Inert at N=1. Fix: unknown type ⇒ retryable backoff (~10 LOC + test). Interim rule: ship a new email type's dispatcher case one release before its producers.

**3. Old-code-on-new-schema windows are governed by nothing.** When `dbVer > embedded`, `CheckSchemaReady` WARNs and returns nil while its own inline comment says it should refuse (migrate.go:1075-1090; the exported godoc at 1045-1055 promises nothing either way). Leader election is version-blind — fixed lock constants, no version affinity (main.go:193) — so during a rolling deploy the OLD replica's scheduler/dispatchers can keep winning the locks and run old billing/dunning code against the new schema for the whole window (unbounded if the rollout wedges at 1-old/1-new). No expand/contract migration discipline exists; `cmd/velox-migrate-safety` measures lock holds and duration, not old-binary compatibility (its main.go:1-37). Worst case is silent data mis-writes, not downtime. Fix: decide + document the skew contract, fix the comment, optionally lint destructive DDL.

**4. Health model masks a wedged scheduler leader at N=2.** Ticks have no per-tick timeout (runner.go:55-90) and each advisory lock lives on a dedicated conn released only by `Release()` or session drop (advisory_lock.go:62-92) — a tick wedged mid-half holds that half's lock indefinitely. The wedged pod's `/health/ready` goes 503 (LB pulls it) but K8s liveness `/health` is an unconditional 200 (router.go:1172-1173, 1447-1449), so nothing restarts it; followers report ready because `onRun` fires on lock-skipped ticks (scheduler.go:249-254). Billing and dunning are separate lock keys (scheduler.go:271-282 vs 485-496), so exactly ONE half stops cluster-wide; the per-pod staleness metric still fires for the pulled pod (runbook.md:32) — alertable, not silent, but only if metrics alerting is deployed. Related N=1 hazard: readiness couples traffic to scheduler pace — a steady-state tick gap >2× interval 503s the sole replica's readiness and self-inflicts a full API outage (router.go:1474-1481; runbook.md:11-12 wires it as the LB check). A restart's long first tick does NOT trigger it (zero lastRun ⇒ "ok: awaiting first run", router.go:1482-1483). Fix: per-half tick timeout; decouple readiness from scheduler staleness; make scheduler liveness a cluster fact persisted in the DB.

**5. Advisory-lock leadership has no fencing: dual-leader overlap duplicates non-CAS side effects (outbound webhook events).** Same failover mechanism as hazard 1; money converges everywhere, but effect-firing tied to idempotent-but-not-CAS writes duplicates — e.g. pause auto-resume: the SKIP LOCKED list tx rolls back before the clear (subscription/postgres.go:1211-1252), `ClearPauseCollection` is unconditionally idempotent with no changed-rows signal (postgres.go:643-679), and each overlapping tick fires `subscription.collection_resumed` via a fresh outbox enqueue (service.go:2337-2358; outbox.go:97-117). The duplicates carry DIFFERENT envelope ids (separate `Enqueue` calls, distinct timestamps), so the documented dedupe-on-event-id guidance (docs/webhooks.md:126) does NOT collapse them — consumers see true semantic duplicates. Rare window; fix is a cheap lock-liveness probe (`SELECT 1` on the lock conn between phases, abort tick on error) rather than full fencing tokens.

**6. Topology self-check has a contention false-pass under restart loops.** `VerifyAdvisoryLockTopology` returns nil when its probe lock is already held, assuming a concurrent booter ("its passing verdict covers this topology too", advisory_lock.go:139-143). Behind a transaction-mode pooler, a first boot that acquires the probe lock and then fails a later probe strands it (the PID-flip path at :145-150 errors without unlocking); every subsequent boot inside the strand window (systemd restart loop, second replica) sees `got=false` ⇒ nil ⇒ starts behind the forbidden pooler, where all five worker locks then strand silently — the exact "week of missing invoices with zero errors" the check exists to prevent (main.go:144-148). Window is minutes-to-an-hour per strand on stock PgBouncer timeouts, but restart loops retry in seconds and the effect persists as long as the topology does. Fix: retry acquisition 3×200ms and fail boot if still contended (~10 lines).

**7. Test-clock catchup is the one scheduler-adjacent worker with NO leader gate.** The queue is an in-process channel, self-declared single-process (catchup.go:63-67: "swap for a Redis or Postgres LISTEN/NOTIFY queue"), and `RecoverInFlight` runs ONCE at boot, re-enqueuing every `status='advancing'` clock with no lease/ownership check (main.go:254-261; testclock/service.go:760-778). Two N=2 shapes: (a) rolling deploy — replica B boots mid-catchup and both run `RunCatchup` concurrently for the same clock (recovery bypasses the `MarkAdvancing` CAS; no lock anywhere in the path). Duplicate invoices CANNOT result (idempotency index; testclock transitions are CAS, postgres.go:224-269) — damage is duplicate test-mode webhook/timeline noise and a possible spurious `internal_failure`, recoverable via Retry. (b) Inverted liveness — the owning replica crashes, the survivor never rescans, and the clock wedges at 'advancing', which has no NON-DESTRUCTIVE exit (Advance needs 'ready', Retry needs 'internal_failure'; Delete works but cascade-cancels pinned subs, postgres.go:129-162). At N=1 restart heals this; at N=2 it's worse. Test-mode only — clock-pinned subs are excluded from all wall-clock paths (subscription/postgres.go:1048-1056). Fix: DB-backed claim (CAS lease or per-clock advisory lock) + periodic rescan on a leader-gated tick; the `CatchupQueue` interface was built for the swap (catchup.go:30-43).

**8. Webhook SSE live tail is a per-process EventBus — at N=2 the dashboard sees a partial, misleadingly-alive tail.** In-process pub/sub, no cross-replica fan-out (eventbus.go:86-160). Frames originate on BOTH replicas: HTTP-path dispatches on whichever replica served the request (payment/settlement.go:436; subscription/handler.go:394; webhook/service.go:704) and dispatcher/retry frames on whichever won that tick's try-lock — not a stable leader — so session affinity does not fully mitigate. The connect-time 50-event snapshot is DB-backed (sse.go:84-92); refresh shows truth. UI-only, no delivery impact. The code names the fix itself: Postgres LISTEN/NOTIFY (sse.go:33-41). Safely deferrable past first cutover.

**9. Password-reset per-address throttle is per-process: cap degrades 3 → 3N/hour.** In-process map, comment explicitly scoped to single-process deploys (user/handler.go:46-48, 51-81, enforced at :353). No enumeration or auth impact; the Redis-backed per-IP limiter still bounds a single attacker fleet-wide. Fix: reuse the `FallbackFailureCounter` shape (lockout.go:165-200), ~half day, deferrable.

**10. Threshold "crossed-but-deferred" notes dedup is a per-process sync.Map: exactly one duplicate audit/timeline row per (sub, period) at N=2.** Per-tick lock release means replicas alternate ticks in normal operation; each replica's map emits the note once, capping duplicates at N−1 per (sub, period) plus one per restart (engine.go:79-81; threshold_scan.go:363-371; scheduler.go:271-282). Operator-visibility artifacts only — the billing decision is invoice-line-watermark-based (threshold_scan.go:316-358). Accept, or dedup against the audit table (~2h).

**11. [SHIPPED 2026-07-06 — ClaimChargeForDunningRetry, same lease column] Sweep vs dunning retry could double-charge via purpose-suffixed idempotency keys.** Fix: the dunning retry path takes the shared per-invoice lease before charging (predicate `payment_status IN ('pending','failed')` — 'unknown' deliberately excluded so ambiguous outcomes wait for the reconciler, which also closes the same-tick N=1 window: the billing half's unknown outcome followed seconds later by the dunning half's due retry); claim-lose maps to ErrTransientSkip (no attempt burn). Pinned by cross-path mutual-exclusion + status-matrix tests. Original finding: The billing and dunning scheduler halves run under SEPARATE advisory locks, so two replicas run them concurrently by design (scheduler.go:218-227). Reachable overlap: a card-less pending invoice enrolled by EnrollStalledForDunning keeps auto_charge_pending=TRUE + payment_status='pending' + an active run; the customer attaches a card; next cycle the auto-charge sweep (purpose "") and a due dunning retry (purpose "dunning_retry") can charge concurrently — the purpose suffix makes their Stripe idempotency keys differ BY CONSTRUCTION (stripe.go:474-477), so Stripe cannot dedupe. Sequential orderings are safe (inline SettleSucceeded → paid → the other path's Finalized guard refuses). Fix shape: dunning's RetryPayment takes the same per-invoice claim CAS (predicate widened to payment_status IN ('pending','failed')) and maps claim-lose to a transient skip (no attempt burn). Needs its own dunning-side tests.

### Verified safe at N=2 — do not re-flag

- **Ingest dedup**: fully DB-level — `ON CONFLICT (tenant_id, livemode, idempotency_key) DO NOTHING`; both doors translate the loser to success (usage/postgres.go:113-115, 169, 182-186; usage/handler.go:171-187; litellm/handler.go:183-197). Batch is one all-or-nothing tx (usage/postgres.go:44-78).
- **Rate limiting with healthy Redis**: shared GCRA buckets, per-limiter namespaces — global limits, no per-replica doubling (ratelimit.go:19-37, 171-208; router.go:1077-1128).
- **API Idempotency-Key middleware**: Postgres-backed reserve-before-act; racing replicas resolve one winner, the loser polls the DB for the winner's response; DB-error fail direction is 409-retry, never re-execute (middleware/idempotency.go:102-131, 203-234, 298-323).
- **Dashboard sessions**: fully DB-backed incl. cross-replica revocation; nothing session-shaped in process memory (session/store.go:42-136).
- **Scheduler lock coverage is COMPLETE**: every scheduler job — 7 reconcilers, auto-charge retry, dunning enrollment, threshold scan, trial expiry, pause resume, RunCycle, credit expiry, cross-mode cleanup — runs inside exactly one of the two advisory-lock scopes (scheduler.go:268-299, 480-508; main.go:193). Followers stay health-green (onRun fires on skipped ticks, scheduler.go:249-254).
- **Invoice generation is exactly-once independent of the leader lock**: `idx_invoices_billing_idempotency`, loser gets 23505 and skips gracefully (subscription/postgres.go:1089-1094).
- **Trial expiry / pause resume double-processing**: SKIP LOCKED candidate fetch with explicit multi-replica rationale + CAS status flips (subscription/postgres.go:2004-2008, 2067-2090, 735/832/886). Credit expiry converges (`ExpireGrantAtomic`, credit/service.go:469-500).
- **All three drainers (email outbox, webhook outbox, webhook retry)**: two-layer design — per-tick advisory-lock gating for efficiency, claim-lease + `FOR UPDATE SKIP LOCKED` + CAS status marks for correctness, disjointness pinned by tests (`TestP5_ConcurrentClaimersDisjoint`, `TestListPendingDeliveries_ConcurrentClaimersDisjoint`). Lease claim atomically bumps attempts, dead claimers' rows re-claimable after lease expiry, marks are CAS'd so a zombie replica can never cause a third send. Webhook event + deliveries + outbox-mark commit in ONE tx (handler-owns-mark), so no crash can mint a duplicate event id (webhook/postgres.go:375-452; email/outbox.go:259-272, 363-391).
- **Livemode/RLS propagation on ctx-detached workers**: both dispatchers pin the row's livemode per-row; scheduler halves fan out per livemode with asserts (email/dispatcher.go:153-158; webhook/dispatcher.go:126-127; scheduler.go:287-310) — pinned by dispatcher_livemode_test.go (d526e94).
- **Concurrent-replica migrations**: the hybrid loop serializes on `LockKeyMigrateHybrid` on a dedicated 1-conn pool; losers poll, re-read `schema_migrations` under the lock, skip applied work, and refuse to step past a dirty peer (migrate.go:125-150, 202-338).
- **Boot gauntlet before listener bind**: migrations → CheckSchemaReady → topology check → ADR-073 RLS posture checks; a misconfigured replica cannot join the fleet (main.go:122-152 vs :288).
- **Leader crash mid-tick**: session advisory locks auto-release on TCP session drop — no zombie-lock recovery needed (advisory_lock.go:58-61); panicking ticks are recover()'d and cannot kill any worker goroutine (runner.go:79-90).
- **Login brute-force protection**: shared Redis counter; lockout DECISION is Postgres `users.locked_until`; degraded mode is a bounded per-process fallback behind a circuit breaker (~threshold×N, never unlimited) (user/lockout.go:15-79, 165-241).
- **Stripe circuit breaker**: per-process but fails safe — breaker-rejected calls are transient/retryable, no invoice failed, no dunning tick; worst case N× half-open probes (payment/breaker/breaker.go:48-56, 89-93).
- **Webhook outbox payload binary-skew**: the dispatcher never interprets the payload — opaque pass-through, old binary delivers new payloads unchanged (webhook/dispatcher.go:118-133). Email `cc` FIELD skew-safe by design (outbox_sender.go:47-56) — the type TAG is hazard 2.
- **Remaining in-process state is benign**: log dedupers, Prometheus gauge-registration guards, test-only mutexes (postgres.go:111; router.go:61). Per-replica metrics counters are standard scrape-and-aggregate.
- One comment-only drift for the HA PR: cmd/velox/main.go:266 says "LockKeyWebhookDispatcher"; the constant is `LockKeyOutboxDispatcher` (advisory_lock.go:17). Cosmetic.

## Downtime = money?

The honest answer: **inside Velox, nothing is ever lost — the entire loss surface is the ingest edge, and it is decided by client retry behavior.** Writes are synchronous, so 2xx means committed; everything pending (billing cycles, dunning steps, auto-charges, outbox rows) is durable and self-heals on restart (docs/ops/backup-considerations.md documents the full resume ledger with validation queries). Inbound Stripe webhooks 5xx during an outage and Stripe retries for ~days, with the payment_unknown reconciler as backstop. What does NOT self-heal:

**1. LiteLLM usage fired while Velox is down is permanently lost at LiteLLM's defaults — this was an open question and is now source-verified (BerriAI/litellm main, read 2026-07-06).** LiteLLM's `generic_api` logger defaults `max_retries=0` and its `async_send_batch` ends with `finally: self.log_queue.clear()` — on ANY send exception the batch is dropped, no re-queue, no durable buffer. Consequences: (a) the repo's three claims that "LiteLLM retries (only) on 5xx" (docs/adr/033-litellm-spend-adapter.md; router.go:1093-1096; litellm/handler.go:104-107) are **false at current LiteLLM defaults** — 5xx raises via `raise_for_status()` into the same drop path, so Postgres-down loses events exactly like process-down; (b) retries are operator-configurable (`max_retries`/`retry_delay`), which shrinks but never closes the window; (c) a LiteLLM-proxy restart loses its in-memory `log_queue` even while Velox is healthy. Two Velox-side aggravators verified in-repo: during a full Postgres outage the request dies at API-key auth, which masks DB failure as **401** (auth/middleware.go:37-47, ADR-026 anti-enumeration) — not a retryable status under any config; and on partial DB failure the spend handler appends persist errors to the envelope and returns **200 OK** unconditionally (litellm/handler.go:138-143, 156) — the comments claiming storage errors propagate as 5xx describe code that does not exist. At the rate-limit sizing floor (>1.7 calls/s, router.go:1094), a 2-minute deploy is ~200+ silently unbilled calls.

**2. The recovery path exists and is clean — but has a hard deadline.** Deterministic per-call idempotency keys (`<litellm_id>:input/:cache_read/:output`, mapper.go:220-243), DB-level `ON CONFLICT DO NOTHING`, and silent-success replay make operator re-POST of LiteLLM's own spend logs a pure gap-filling replay with preserved original timestamps (mapper.go:139-148). The deadline: by documented design (usage/service.go:214-226, "KNOWN BEHAVIOR"), an event whose timestamp falls in an already-finalized period is stored but **effectively unbilled** — aggregation is strictly timestamp-windowed (usage/postgres.go:344-393) and nothing true-ups closed periods. Replay after finalize preserves the audit ledger but not the revenue; recovery then means a manual invoice adjustment. A downtime window straddling period close, discovered late, is the permanent-loss case. The late-event counter named in that comment is the smallest visibility fix.

**3. Direct API clients**: delayed if they retry non-2xx, lost if they don't. The retry contract ("non-2xx means not recorded — retry; replays are dedup-safe") is not yet stated in the ingest docs. One narrow pre-existing N=1 hazard, N=2-neutral: a **keyless** single-event POST whose response died to a hard crash (kill -9 between commit and response; graceful deploys drain 30s) double-bills on retry — keyless events have "no dedup line of defense" (usage/postgres.go:48), NULLs never conflict. Batch is immune even keyless (one tx); LiteLLM is immune (always keyed). Docs-first fix; do not reject keyless (Stripe meter events accept keyless too).

**4. Shutdown mechanics**: worst-case drain is ~11min (HTTP 30s → test-clock catchup up to 10min on a `context.Background()` ctx that shutdown cannot cut short, catchup.go:57/147, main.go:294-328 — the comment at main.go:308-309 claiming otherwise is wrong → workers 30s), exceeding K8s' default 30s grace; SIGKILL mid-anything is designed-for (outbox rows pending, catchup resumes at boot). Set grace ≥ ~75s and cap the shutdown catchup wait. Unclean kills also cost bounded at-least-once duplicates: ≤5 emails + ≤10 webhook POSTs re-sent once each, webhooks with the SAME event id (receivers dedupe; SMTP can't), and one of 15 attempts burned per claimed in-flight row — deliberate, so a crash-looping row still converges to the DLQ (email/outbox.go:189-196; webhook/postgres.go:436-446).

**5. Restore-to-earlier-point and key loss** (see SPOF register): duplicate outbox re-fires + lost open-period usage unless PITR; a DB backup without `VELOX_ENCRYPTION_KEY` is unrecoverable.

## Refuted-hazard appendix

One line each, so future audits don't re-flag. (No whole finding was refuted this audit; these are refuted candidate-hazards and refuted sub-claims.)

- **Test-clock double-run duplicates invoices** — refuted: `idx_invoices_billing_idempotency` (subscription/postgres.go:1089-1095) + CAS clock transitions (testclock/postgres.go:224-269); loser skips gracefully.
- **Concurrent credit-applies diverge the auto-charge idempotency key** — refuted: `ApplyToInvoiceAtomic` serializes under FOR UPDATE, at most one racing apply bumps `updated_at` (credit/postgres.go:609-792); the real divergence path is `UpdatePayment` after a failed/unknown outcome (hazard 1).
- **"Postgres-down is safer than process-down for LiteLLM"** — refuted: `_should_retry_exception` treats TransportError and 5xx identically; both hit the same drop path.
- **Fast ticks overlap at N=1** — refuted: ticks are strictly sequential per replica (runner.go:64-73); dual-leader overlap requires N≥2.
- **Envelope-id dedup absorbs dual-leader duplicate events** — refuted: separate `Enqueue` calls mint distinct ids (webhook/outbox.go:109), so docs/webhooks.md:126 guidance does not collapse them.
- **A wedged 'advancing' clock has no operator exit** — overstated: `Delete` is not status-gated (testclock/postgres.go:129-145) — destructive (cascade-cancels pinned subs) but available.
- **Restart-with-backlog first tick trips readiness** — refuted: zero lastRun reports "ok: awaiting first run" (router.go:1482-1483).
- **Scheduler leadership "flaps" and re-emits threshold notes per flap** — refuted: no sticky leadership exists (per-tick lock); each replica's sync.Map bounds emission to once per process per (sub, period).
- **Dashboard sessions / idempotency cache split-brain at N=2** — refuted: both Postgres-backed (session/store.go:42-136; middleware/idempotency.go:203-234).
- **Rate limits double at N=2** — refuted for all Redis-backed limiters (shared GCRA); true only for the in-process reset throttle (hazard 9).
- **Outbox dual-drain double-sends** — refuted: SKIP LOCKED claims + lease + CAS marks, pinned by concurrent-claimer tests.
- **Racing boot migrations at N=2 corrupt schema state** — refuted: serialized on `LockKeyMigrateHybrid` with under-lock re-read and dirty refusal (migrate.go:202-338).
- **Redis is a boot SPOF** — refuted: boot warns and continues without Redis (router.go:1049-1059); it is a prod *runtime* availability SPOF instead (fail-closed limiters).
- **Batch ingest can commit a prefix and double-bill on retry** — refuted: single all-or-nothing tx (usage/postgres.go:44-78).
- **Follower replicas report degraded scheduler health at N=2** — refuted: `onRun` fires on lock-skipped ticks (scheduler.go:249-254).
- **"LiteLLM retries on 5xx" (as the repo claims in three places)** — refuted at LiteLLM defaults (`max_retries=0`, batch dropped on any exception); the repo comments need correcting, not the finding.
- **Drainer downtime loses queued messages** — refuted: durable rows, claim-at-lease, at-least-once with bounded duplicates; only decay case is time-boxed links (>1h outage).
