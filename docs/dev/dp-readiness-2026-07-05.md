## metering-ingest — gaps-manageable

### fix-now

- **Hardcoded 100 req/min rate limit caps the ingest path — the LiteLLM wedge integration 429s at pilot traffic** *(effort: hours)*
  - DP impact: ADR-033: LiteLLM POSTs one callback per LLM call. Any real AI-infra traffic (>1.7 calls/s sustained) exhausts the GCRA bucket within the first hour of wiring the proxy; LiteLLM retries only on 5xx (ADR-033 explicitly relies on this), so 429'd events are silently dropped = permanently lost revenue data with no counter. Even disciplined /batch users cap at 100 batches/min. This fires on day 1 of the pilot, during the exact demo the wedge is sold on.
  - Evidence: internal/api/router.go:1021 `mw.NewRateLimiter(rdb, "general", 100, time.Minute)` applied to all /v1 at router.go:1219 (covers /usage-events at :1300-1301 and /integrations/litellm/spend at :1308-1309); grep of internal/config shows NO env override for the rate; MANUAL_TEST.md FLOW X3 documents 100/min as intended behavior; docs/adr/033-litellm-spend-adapter.md:19 'LiteLLM POSTs each call'.
  - Peer standard: Orb: 10 req/s + 2,000 events/min in TEST mode alone; production 'tens of millions of events per day', dedicated-capacity talk only above 10k events/min (verified docs.withorb.com/events-and-metrics/event-ingestion). Stripe meter events: 1,000 calls/s live mode, 10k events/s via v2 streams (verified docs.stripe.com). Velox's live limit is ~6x below Orb's test-mode limit.
- **Usage Events dashboard dimension filter is dead — server silently ignores the param and shows unfiltered data as filtered** *(effort: day)*
  - DP impact: Filtering by dimension (e.g. model=gpt-4o) is the first move a DP engineer makes when validating token pricing or debugging 'why didn't my event bill'. The input renders, accepts a value, and the stat cards + event log re-render with UNfiltered totals presented as filtered — actively misleading during pilot pricing validation, worse than the filter not existing.
  - Evidence: web-v2/src/pages/UsageEvents.tsx:113 pushes `dimensions=${encodeURIComponent(filterDim)}` into both the list and aggregate queries; internal/usage/handler.go:207-240 parseListFilter reads only customer_id/meter_id/from/to/limit/offset/after; internal/usage/store.go:48-68 ListFilter has no Dimensions field; postgres.go:589-618 buildUsageWhere has no JSONB clause; repo-wide grep for a `dimensions` query-param reader returns nothing. GIN index on usage_events.properties already exists (migration 0062) so a `properties @> $n` clause is cheap. MANUAL_TEST FLOW U3 only claims customer/date filters — the UI shipped ahead of the server.
  - Peer standard: Lago's events list and Orb's event explorer both filter/inspect by event properties (PLAUSIBLE — not fetched this session; Lago events-list doc verified to exist as a first-class debugging surface).
- **Idempotent replays are reported as failures on the public API (409 / HTTP 206 + error rows), inconsistent with the LiteLLM door** *(effort: hours)*
  - DP impact: A network-level retry of a fully-successful 1000-event batch returns HTTP 206 with 1000 'duplicate key' error strings — the DP's pipeline alerting fires on healthy at-least-once delivery, and their engineers must string-parse errors to separate benign dupes from real failures. Single-event replay 409s with no fetch-original. Hit in week 1 by anyone with standard retry middleware.
  - Evidence: internal/api/respond/errors.go:57-58 (ErrDuplicateKey → 409 Conflict); MANUAL_TEST FLOW B13: 'Replay same idempotency_key → 409 Conflict … no fetch-original-on-replay'; internal/usage/handler.go:329-345 batch returns 206 with error strings; contrast internal/integrations/litellm/handler.go:193 which treats ErrDuplicateKey as silent success — two front doors, two contracts. Unique scope: (tenant_id, livemode, idempotency_key), migration 0020_test_mode.up.sql:74.
  - Peer standard: Orb: 'duplicates are never processed within the account grace period' — deduplication is success, not an error (verified docs.withorb.com/events-and-metrics/event-ingestion). Fix shape: count dupes as a `deduplicated` field and return 2xx, mirroring the LiteLLM envelope.
### pre-frame

- **No ingest-error observability: rejected events vanish into a response body nobody reads (LiteLLM path) — no rejected-events metric, dead-letter, or UI** *(effort: week+)*
  - DP impact: One misconfigured call site (developer forgets user=<external_customer_id> on LiteLLM calls, or a meter key typo) = those calls silently never bill. docs/integrations/litellm.md:64 says 'monitor errors[], not the HTTP status code' — but LiteLLM's generic logger discards response bodies, so errors[] is write-only. Discovery happens at month-end when the invoice is short. Near-certain in 30 days: DPs onboard new services/teams to the proxy continuously.
  - Evidence: internal/integrations/litellm/handler.go:148-155 — per-row failures go to errors[] + one slog.Warn only; internal/api/middleware/metrics.go has velox_usage_events_ingested_total but grep confirms no rejected/dropped counter anywhere in internal/usage or internal/integrations; no dead-letter table in migrations; docs/ops/grafana-dashboard.json has no rejection panel.
  - Peer standard: Lago ships an events list/debugger UI with per-event warnings ('event code is not related to an existing billable metric', 'property used for aggregation is not sent') plus alerting on ingestion errors (verified getlago.com/docs/guide/events/events-list + ingesting-usage). Stripe v2 meter events pair sync validation with async meter error reports.
- **'Ingested but will never bill' is silent: unpriced/unclaimed usage and late events into finalized periods produce only server-log WARNs** *(effort: day)*
  - DP impact: Two paths a DP hits in the first month: (1) a provider ships a new model, events arrive with a model dimension no pricing rule matches → if the meter has no default binding the revenue is dropped with a WARN nobody tails (COGS side has an honesty counter — margin endpoint's unresolved_events — but the REVENUE side has no equivalent); (2) a pipeline outage spanning a period close → late events are accepted then never billed, because invoices auto-finalize at cycle close with zero grace window. The in-code comment itself says the true-up policy is a 'Deferred decision (no design partner yet)' — that trigger is now firing.
  - Evidence: internal/billing/engine.go:2757-2764 (unclaimed usage with no default binding → slog.Warn 'not billed', continue); internal/usage/service.go:194-206 KNOWN BEHAVIOR block (late events accepted, effectively unbilled, deferred decision); no unbilled-usage metric or report exists (grep). Not in the ADR-README follow-up table — untracked.
  - Peer standard: Metronome: 24h default grace period before invoice finalization to absorb late usage, void+regenerate to absorb backdated corrections (verified docs.metronome.com/guides/get-started/core-concepts/how-invoicing-works). Orb: backdated/backfilled events with automatic invoice updates. Lago: per-event warnings on unmatched events.
- **No client SDKs for the event API — raw HTTP only** *(effort: week+)*
  - DP impact: The LiteLLM path is genuinely SDK-free, but any usage NOT flowing through the proxy (custom pipelines, non-LLM meters like GPU-hours or storage, backfills) needs hand-rolled HTTP + retry + idempotency-key management in week 1 — and every rough edge above (409-on-replay, 206 semantics) lands directly on the DP's engineers instead of being absorbed by a client library.
  - Evidence: No sdk/ or clients/ directory in repo root (ls output); README.md:15,66,88 markets 'no SDK' only as a LiteLLM-path virtue. Usage-events routes ARE in the OpenAPI slice (web-v2/src/lib/gen/schemas/postV1UsageEventsBody.ts et al. are generated), so an openapi-generated Python/TS ingest client is a ~day mitigation to offer the DP.
  - Peer standard: Orb, Metronome, Lago, and Stripe all ship official Python + Node SDKs for event ingestion (PLAUSIBLE — industry-known, not fetched this session).
### defer

- **Batch ceilings: 1000 events/request under a 1MB body cap, processed as sequential per-event transactions** *(effort: week+)*
  - DP impact: 16-dimension AI events run ~0.4-0.6KB each; a 1000-event batch approaches the 1MB MaxBytesReader cap, and each batch executes ~3 queries per event sequentially (customer resolve + meter resolve + insert tx). Fine at pilot volumes once the rate limiter (gap 1) is fixed — the service layer benches 50k events/s — but the HTTP door becomes the bottleneck at sustained >1k events/s. Named trigger: DP reports sustained high-rate direct-API ingestion or 413s on max batches.
  - Evidence: internal/usage/handler.go:314-316 (1000 cap); internal/api/router.go:1081-1084 (1MB MaxBytesReader on all routes); internal/usage/service.go:228-245 BatchIngest loops one store.Ingest tx per event; handler.go:320-327 resolves per event with no caching; cmd/velox-bench/main.go validates 50k events/s through the service layer (FLOW B13).
  - Peer standard: Orb caps at 500 events/batch (Velox is 2x higher — fine); Stripe offers meter event streams for 10k+/s. Batch size itself is at-peer; the per-event tx loop and body cap are the eventual ceilings.
### Strengths above peer standard (pitch these)

- LiteLLM proxy drop-in with zero glue code: one YAML callback → per-call events split into disjoint token roles (input / output / cache_read) on ONE tokens meter, with model/model_raw/provider/team_id dimensions, replay-deduped as silent success, and misconfig never 5xxes into retry storms (ADR-033/044; internal/integrations/litellm/). No first-party equivalent at Orb/Metronome/Lago/Stripe (PLAUSIBLE) — they'd need custom pipeline code for the same.
- Per-event provider COGS stamped at ingest inside the INSERT itself (zero extra round trips) + in-app per-customer/per-model margin with honesty semantics: 'table' / 'not_applicable' / NULL-unresolved plus an unresolved_events counter (ADR-079; internal/usage/postgres.go:58-93, FLOW M1). Peers price revenue; none stamps operator COGS per event natively (PLAUSIBLE).
- Dimension-matrix pricing on one meter (ADR-044): {model × token_type} rules with deterministic priority+claim resolution, vs Stripe's one-meter-per-dimension-combination sprawl — the README's core pitch, and the aggregation SQL (including double-count-proof last_ever claims) is real (internal/usage/postgres.go:427-587).
- Synchronous ingest validation with actionable 422s ('customer "x" not found', 'meter "y" not found', quantity magnitude, dimension scalar/16-key caps) — matches Stripe v2's sync-validation shape and beats Lago's accept-async-then-warn default, so most bad events fail loudly at the front door instead of silently at billing.
- Test-clock-aware ingestion: events land on the pinned customer's simulated timeline (frozen_time default + future-gate against simulated now, internal/usage/service.go:48-60,171-191), so the full ingest→advance→hybrid-invoice demo runs in minutes; live-mode path pays zero extra queries for it.
- Money-honest metering details peers get wrong: zero-quantity events stay 0 (the silent coerce-to-1 money bug was found and killed, service.go:153-161), NUMERIC(38,12) decimal quantities end-to-end, negative corrections allowed, overflow gated as clean 422.
- Self-host ops baseline in-repo: Prometheus metrics including velox_usage_events_ingested_total, shipped Grafana dashboard JSON, OTel tracing, Stripe-convention X-RateLimit-*/Retry-After headers, cursor pagination on the highest-write table, and 5-minute streaming CSV usage exports (router.go:1188-1209).
- Capacity story is bench-verified, not claimed: cmd/velox-bench drives the real service path (transactions, realistic 80-combination dimension cardinality) at 50k events/s on local Postgres (FLOW B13) — once the HTTP rate limiter is made configurable, the engine underneath already has pilot headroom.

## pricing-catalog — gaps-manageable

### fix-now

- **Free/included allowance is inexpressible — zero-price tiers and zero flat rates are rejected at authoring** *(effort: hours)*
  - DP impact: Day 1-3, first config session: almost every AI-infra pricing page has 'includes N free tokens/requests/GPU-minutes'. Authoring a graduated rule with a $0 first tier returns 422 'unit price must be greater than 0'. The only workaround is manual one-time credit grants per customer per month (credit.GrantInput has no recurrence), which finance won't accept as 'free tier'.
  - Evidence: internal/pricing/service.go:186-188 (flat: `!flat.IsPositive()` → 'unit price must be greater than 0') and :196-198 (each graduated tier: `!tier.UnitAmountCents.IsPositive()` rejected). ComputeAmountCents itself accepts zero rates (internal/domain/pricing.go:217,243 check IsNegative only) — the block is authoring validation only, so the relax is hours.
  - Peer standard: Stripe explicitly supports a zero-fee first tier in graduated pricing (docs.stripe.com/subscriptions/pricing-models/tiered-pricing, verified this session); Lago has free_units on package/percentage charges and Orb has included-unit allocations (PLAUSIBLE, not fetched this session).
- **Usage matching no pricing rule is silently unbilled at cycle close — and the shipped recipes create exactly this trap for every newly launched model** *(effort: day)*
  - DP impact: Week 2-3: a provider ships a new model (monthly cadence in AI), LiteLLM ingests events with the new `model` dimension, no (model, token_type) rule matches, and the invoice closes WITHOUT that revenue — the only trace is a server-log WARN on a box the DP's finance team never reads. Preview/customer-usage APIs do carry a warning string, but nothing proactive (no invoice attention, no webhook, no dashboard badge) fires at close.
  - Evidence: internal/billing/engine.go:2757-2764 (unclaimed bucket with no meter default → slog.Warn '…not billed' + continue); internal/billing/preview.go:244 and internal/usage/customer_usage.go:405 (warning surfaced only if the operator looks). The catch-all primitive EXISTS — a `dimension_match: {}` priority-0 rule matches every event (internal/domain/pricing.go:155-160) — but internal/recipe/recipes/anthropic_style.yaml and openai_style.yaml ship zero catch-all rules and the recipe meters carry no default binding; the meter-level default (meters.rating_rule_version_id) can only be set at meter CREATE — MeterRoutes has no PATCH (internal/pricing/handler.go:40-46; store UpdateMeter is unwired, no service/handler caller).
  - Peer standard: Orb matrix pricing: an event that matches no dimension value 'resorts to the default_unit_amount' — a required default rate, so unmatched usage is always priced (docs.withorb.com/product-catalog/price-configuration, verified this session).
- **Recipe catalogs are stale and numerically wrong — the first thing an AI-infra evaluator opens** *(effort: hours)*

### pre-frame

- **No bulk pricing import or rate-card editor — a real pricing sheet is 2 API objects per (model, role) cell** *(effort: week+)*
  - DP impact: Week 1 onboarding: recipes only parameterize currency/plan_name/plan_code ('the rest is fixed for v1'), so the DP's actual model lineup means hand-authoring. 20 models x 5 token roles = 100 rating rules + 100 meter pricing rules = ~200 API calls or ~200 passes through the one-rule-per-dialog UI. An engineering lead will script it in an afternoon (API is clean and idempotent-upsert on pricing rules), but it's the first hands-on impression and reads far behind Metronome. State it proactively: 'config-as-code today, rate-card import on the roadmap'.
  - Evidence: internal/recipe/recipes/anthropic_style.yaml `overridable:` block (only currency/plan_name/plan_code); web-v2/src/pages/MeterDetail.tsx:464-508 (CreatePricingRuleDialog — one rule per dialog); internal/pricing/handler.go:57-62 (one rating rule per POST, no batch route); no import command or endpoint anywhere in internal/pricing/. Note ADR-079's tracked import trigger covers COST rates only, not price rules.
  - Peer standard: Metronome rate cards are the centralized pricing entity with an addRates API for scheduling/adding rates in bulk (docs.metronome.com/guides/implement-metronome/core-concepts/create-manage-rate-cards, verified this session); Orb ships a price-matrix editing UI (PLAUSIBLE).
- **No future-dated price changes — a publish takes effect at each customer's NEXT period open, so 'new rates effective Oct 1' is inexpressible under anniversary billing** *(effort: week+)*
  - DP impact: Day ~20-30, first repricing conversation: the DP announces a price change effective a calendar date. In Velox a version's created_at IS its effectivity (ADR-070 decision 4, deliberate), so the operator must publish on the announcement day and rates then roll per-customer as each anniversary period opens — customers reprice on different days, and there is no way to stage a publish ahead of time. Correct and audit-clean, but it surprises anyone trained on Metronome/Orb scheduled rate changes; pre-frame it as 'rolling at period boundary, by design — scheduled effectivity on the roadmap'.
  - Evidence: docs/adr/070-price-change-semantics.md decision 4 ('no separate effective_from column; a version's creation IS its effectivity') + decision 2 ('a publish takes effect at the next period open'); internal/pricing/postgres.go:112 + store.go:19-25 (GetRuleByKeyAsOf keyed on created_at <= periodStart).
  - Peer standard: Metronome: 'to schedule a rate change through the API, make a POST request to /contract-pricing/rate-cards/addRates' — future-dated rate scheduling is first-class (docs.metronome.com, verified this session); Orb price intervals support scheduled changes (PLAUSIBLE).
- **No volume-mode tiering and no per-tier flat fees** *(effort: day)*
  - DP impact: Week 1 config, IF their sheet uses them: 'past 100M tokens, ALL tokens bill at $2.50/M' (volume/all-units repricing) has no mode — only graduated (marginal brackets), flat, and package exist. Per-tier flat fees ('$99 covers the first 20k, then $0.005/unit' — Stripe's own docs example) are also inexpressible since RatingTier carries only up_to + unit_amount_cents. Ask for the DP's actual price sheet before the pilot; if volume discounts appear, this jumps to fix-now (ComputeAmountCents is a clean seam — one new mode).
  - Evidence: internal/domain/pricing.go:13-17 (PricingMode = flat|graduated|package only; `grep -rin volume internal/domain/pricing.go internal/pricing/` → no hits) and :19-29 (RatingTier has no flat_amount field).
  - Peer standard: Stripe tiers_mode supports both 'graduated' and 'volume', and tiers carry flat_amount + unit_amount (docs.stripe.com/subscriptions/pricing-models/tiered-pricing, verified this session); Lago ships a volume charge model and per-tier flat fees on graduated (getlago.com docs, verified this session).

### Strengths above peer standard (pitch these)

- Full Anthropic cache-economics parity in the metering model (ADR-044): one `tokens` meter, `token_type` role dimension with cache_read / cache_write_5m / cache_write_1h split — three distinct cache prices that a boolean 'cached' flag (and most peers' quickstarts) cannot express; verified in-repo against Lago/OpenMeter/Orb convergence with sources in the ADR.
- Exact sub-cent per-token rates end-to-end (ADR-045): per-unit rates are arbitrary-precision decimals (internal/domain/pricing.go:19-29), quantities NUMERIC(38,12) (internal/domain/usage.go:19), line amounts int64 cents with banker's rounding — $3.00/1M-token pricing bills linearly with zero per-million rounding drift; Stripe unit_amount_decimal shape, without Stripe's decimal-place cap (PLAUSIBLE on the cap detail).
- Audit-grade price-change semantics (ADR-070): immutable rating-rule versions, one shared resolver pinning the whole period at period-open across close/cancel/threshold/preview (preview == invoice by construction, internal/billing/preview.go + engine.go:2698-2703), overrides that follow the rule across version publishes, a currency-change guard that refuses to silently re-denominate negotiated overrides (internal/pricing/service.go:75-101), and loud-fail resolution (a DB blip can never silently bill list price). Peers schedule rate changes; none of Orb/Metronome/Lago pitch this level of resolution coherence.
- In-app COGS + per-customer margin (ADR-079): provider cost tables with cost stamped at ingest (immune to later rate edits), per-model margin with an honest unattributed_revenue bucket instead of heuristic allocation — no peer billing engine (Orb/Metronome/Lago/Stripe) ships provider-cost margin natively (PLAUSIBLE); for an AI-infra DP this answers 'which customers lose us money' inside the billing tool.
- Authoring-time money guardrails above peer baseline: full tier-shape validation at create (non-monotonic tables and bounded final tiers rejected before they can block an invoice at close — internal/pricing/service.go:175-228, same contract enforced on overrides), and Stripe-parity plan immutability (billing-affecting fields locked while live subs reference the plan, internal/pricing/service.go:383-425).
- Recipes as one-click provider-shaped catalogs (3 shipping: Anthropic/OpenAI token matrix, Replicate GPU-time) that install meters + rules + plan + dunning policy + webhooks atomically with rollback, and reinstall by adopting existing objects by natural key so operator edits are never clobbered (ADR-070 consequences; FLOW R1-R5) — provided the catalog-staleness fix-now lands first.
- Per-customer negotiated pricing with provenance: overrides carry a reason field, append-only effectivity windows resolved as-of period open (mid-period edits can never retro-reprice an open period), soft-deactivate keeps full audit history (internal/pricing/override.go).

### pre-frame

- **No backdated or future-dated subscription start, no billing-cycle-anchor control** *(effort: week+)*
  - DP impact: Day 1 of migration: onboarding existing customers mid-contract, the DP cannot preserve existing renewal dates.
