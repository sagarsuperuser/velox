# Audit Log Retention Guide

Companion to [backup-recovery.md](./backup-recovery.md) and
[runbook.md](./runbook.md). Backup/restore answers "can we get the data
back?"; this doc answers "how long do we keep the evidence of who did
what, and where do we put it after that?"

Velox writes one row to `audit_log` for every successful mutating
request. The table is the source of truth for SOC-2 / GDPR / financial
audits — losing rows is a compliance gap, but keeping every row
forever in the hot table makes queries slow and storage expensive.
This guide covers what's recorded, how long to keep it, and how to
prune + archive without taking the table offline.

## What's in the audit log

### Schema

Defined in `internal/platform/migrate/sql/0001_schema.up.sql` (table
creation), `0010_audit_fail_closed.up.sql` (per-tenant fail-closed
opt-in lives on `tenant_settings`), `0011_audit_append_only.up.sql`
(immutability trigger), and `0030_audit_request_context.up.sql`
(`request_id` column + index dedup).

| Column | Type | Captures |
|---|---|---|
| `id` | `TEXT PRIMARY KEY` | Random `vlx_aud_<24-hex>` per row, generated by Postgres `gen_random_bytes`. |
| `tenant_id` | `TEXT NOT NULL` | RLS-scoped — every read goes through `db.BeginTx(ctx, postgres.TxTenant, tenantID)`. FK to `tenants(id) ON DELETE RESTRICT` so a tenant cannot be removed while their evidence still exists. |
| `actor_type` | `TEXT NOT NULL CHECK IN ('api_key','user','system')` | Three-valued. `user` is reserved for the WorkOS-backed UI login that lands when the dashboard ships first-class auth; today every operator action is `api_key`. `system` fires when the scheduler / billing engine / dunning loop write rows on their own behalf. |
| `actor_id` | `TEXT NOT NULL` | The API-key id, future user id, or the literal string `system`. The audit query handler `LEFT JOIN`s `api_keys` on this column to surface a human-readable key name in the dashboard so operators see "Production" instead of `vlx_secret_live_…`. |
| `action` | `TEXT NOT NULL` | A verb like `create`, `update`, `cancel`, `finalize`, `rotate` (full list in `internal/domain/audit.go`). Path-derived for catch-all middleware writes (`/subscriptions/{id}/cancel` becomes `cancel`); explicit handler writes use the `domain.AuditAction*` constants directly. |
| `resource_type` | `TEXT NOT NULL` | Singular snake_case: `subscription`, `invoice`, `coupon`, `customer`, `credit`, `credit_note`, `customer_coupon_assignment`, `plan_migration`, `api_key`, `webhook_endpoint`, `rating_rule`, `usage_event`, `meter`, `billing_profile`, `bulk_action`. The catch-all middleware normalizes via `canonicalResourceType`; explicit writes pass the canonical form directly. |
| `resource_id` | `TEXT NOT NULL` | The Velox id of the affected row. `LEFT JOIN`-friendly even if the underlying row is later soft-deleted, because the audit table has no FK on this column — by design, evidence outlives data. |
| `resource_label` | `TEXT DEFAULT ''` | A human-readable label captured at write time (invoice number, customer display name, plan name) so dashboards don't have to re-resolve names that may have changed since. Extracted by `extractLabel()` in the catch-all middleware from a small set of well-known JSON fields. |
| `metadata` | `JSONB NOT NULL DEFAULT '{}'` | Free-form structured context. Handlers populate this with the before/after diff, the originating request path (catch-all writes record `{"path":"/v1/…"}`), and any business-relevant context — e.g. `subscription.proration_failed` carries the reason; `bulk_action.completed` carries `customer_filter` + counters; `plan.migration_committed` carries `from_plan_id` / `to_plan_id` / `totals`. Use the `metadata` JSONB to answer "what changed" without re-running history. |
| `ip_address` | `TEXT` | Client IP extracted by `audit.ExtractClientIP` (X-Forwarded-For first, X-Real-IP, then `RemoteAddr`). Stored as text rather than `inet` so middleware can record exotic forwarding headers verbatim; nullable because background system writes have no client IP. |
| `request_id` | `TEXT` | The chi per-request UUID (`chimw.GetReqID`). Joins back to structured server logs for any incident, and matches the `Velox-Request-Id` header echoed to callers — give a customer this value during a support ticket and ops can pull the exact server log. Added by migration `0030_audit_request_context`. |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` | UTC. Indexed `(tenant_id, created_at DESC)` via `idx_audit_log_created` for the default reverse-chronological query. |

Indexes:

- `idx_audit_log_resource (tenant_id, resource_type, resource_id)` — drives "show every change to subscription X".
- `idx_audit_log_created (tenant_id, created_at DESC)` — drives the reverse-chronological list and the catch-all "last 30 days" query.

A duplicate `idx_audit_log_tenant` on the same `(tenant_id, created_at DESC)` keys was dropped in migration `0030_audit_request_context.up.sql` — every audit insert now writes one fewer index entry.

### Append-only enforcement

Migration `0011_audit_append_only.up.sql` installs a `BEFORE UPDATE OR DELETE` trigger named `audit_log_immutable_trg` on `audit_log`:

```sql
CREATE TRIGGER audit_log_immutable_trg
BEFORE UPDATE OR DELETE ON audit_log
FOR EACH ROW EXECUTE FUNCTION audit_log_immutable();
```

Any `UPDATE` or `DELETE` against the table — from the API, an ORM, an `IDE`, or a careless `psql` session — raises `audit_log is append-only; UPDATE/DELETE is not permitted`. The error message hints at the legitimate retention path: a DB admin temporarily drops the trigger, runs the prune, and recreates the trigger. That DDL is itself captured by the Postgres statement log, so the audit chain is preserved one level above the table.

### What events are recorded today

Every successful 2xx mutating HTTP request (POST / PUT / PATCH / DELETE) outside `/v1/bootstrap`, `/v1/webhooks/stripe`, `/health`, and `/metrics` produces an audit row, written by `internal/api/middleware/audit.go::AuditLog`. The middleware buffers the response, attempts the audit insert, and either flushes the handler's response (fail-open) or replaces it with `503 audit_error` (fail-closed) based on `tenant_settings.audit_fail_closed`.

In addition, individual handlers call `auditLogger.Log(...)` explicitly when they want to record a richer `metadata` payload than the catch-all path can derive from the URL. These handler-explicit writes set `audit.MarkHandled(ctx)` to suppress the catch-all duplicate. Concrete event types currently emitted (grep-confirmed):

| Source | `action` | `resource_type` | Notes |
|---|---|---|---|
| `internal/credit/handler.go` | `grant`, `update`, `revoke`, `expire`, `adjust` | `credit` | Credit ledger operations. |
| `internal/coupon/handler.go` | `create`, `update`, `archive`, `unarchive`, `redeem` | `coupon` | Plus `create` / `revoke` on `customer_coupon_assignment` for per-customer attachments. |
| `internal/subscription/handler.go` | `cancel`, `update`, `subscription.proration_failed`, `subscription.item_updated`, `subscription.plan_changed` | `subscription` | The dotted `subscription.*` actions are intentional — they collide with no path-derived verb and stay searchable as a family. |
| `internal/invoice/handler.go` | `finalize`, `void`, `rotate`, `refund`, `apply_coupon` | `invoice` | `rotate` covers hosted-invoice public-token rotation. |
| `internal/creditnote/handler.go` | `credit_note.issued` | `credit_note` | Issued via the credit-note service; finalisation rolls into the invoice trail. |
| `internal/customer/gdpr.go` | `gdpr.delete` | `customer` | Right-to-erasure. Records the original `display_name` in metadata so evidence survives the anonymisation. |
| `internal/planmigration/service.go` | `plan.migration_committed`, `subscription.plan_changed` | `plan_migration`, `subscription` | One cohort summary plus per-customer entries. |
| `internal/bulkaction/service.go` | `bulk_action.completed`, `customer.coupon_assigned`, `subscription.cancel_scheduled` | `bulk_action`, `customer`, `subscription` | One cohort summary plus per-customer entries; `bulk_action_id` is recorded in the per-customer metadata so CS reps can answer "why was my coupon applied?" without joining tables. |
| `internal/api/middleware/audit.go` (catch-all) | path-derived: `create`, `update`, `delete`, `activate`, `cancel`, `pause`, `resume`, `finalize`, `void`, `archive`, `unarchive`, `rotate`, `change_plan`, etc. | path-derived (`subscription`, `invoice`, …) | Anything not handled explicitly. Records `metadata.path` and the response-extracted `resource_label`. |

### What's NOT recorded today

A few categories deliberately bypass the catch-all middleware:

- **`/v1/bootstrap`** — the one-shot first-tenant creation. Audit infrastructure isn't yet bound to a tenant id at the moment bootstrap runs. The bootstrap event is captured as a side-effect in `tenants.created_at` plus the application log.
- **Inbound `/v1/webhooks/stripe`** — Stripe is the actor here, not an operator. `stripe_webhook_events` is the source of truth for those, with its own dedupe + replay machinery (migrations `0058_webhook_event_replay`).
- **`/health`, `/metrics`** — never operator actions.
- **GET requests** — read traffic isn't audited. Reads can be high-volume and reading is generally not the compliance-critical surface; a SOC-2 audit cares about who *changed* customer data, not who looked at it. If your regime requires read-side audit (some healthcare and government tenants do), wire it in at the application middleware layer, not the database — see "Adding new event types" below.
- **Failed mutations** — the middleware checks `buf.status >= 200 && < 300` before writing. A 4xx / 5xx response means the business mutation didn't commit, so there's nothing to audit. Auth failures, validation errors, and rate-limit rejections are not in `audit_log`; they're in the application log and (for repeated auth failure) in the `api_keys.last_used_at` / future `api_key_events` table.

### Adding new event types

Two surfaces; pick by where the event semantically lives.

1. **A handler that wants richer `metadata` than the URL can describe** — call `auditLogger.Log(ctx, tenantID, action, resourceType, resourceID, map[string]any{...})` directly in the handler after the business write commits. Use the `domain.AuditAction*` constants when an existing verb fits; introduce a new `package.action` form (e.g. `subscription.plan_changed`) when you want a distinct query bucket. The handler-explicit call automatically suppresses the catch-all middleware via `audit.MarkHandled(ctx)`.
2. **A new mutating endpoint** — if the path follows the `/v1/<resource>/{id}/<verb>` convention and the verb is in the `parseAuditPath` switch in `internal/api/middleware/audit.go`, the catch-all picks it up with no change. New verbs need a one-line addition to the switch. Anything outside the convention (action-style endpoints, multi-resource mutations) should write its own audit row.

Background workers (the scheduler, billing engine, dunning loop) call `auditLogger.Log` directly with `actorType="system"`. The middleware path doesn't apply because there's no HTTP request.

## Compliance baseline retention

There is no single answer. Velox holds financial data for B2B SaaS tenants whose own customers may be in healthcare, finance, or EU jurisdictions. The guidance below is the regime-by-regime baseline; tenants pick the longest window any of their regimes demands.

### SOC 2 Type 2

**Recommended: 12 months minimum during the audit period; 7 years if the same data also serves SOX or financial-services audits.**

Reasoning: SOC 2 Type 2 audits cover a **review period** — typically the prior 12 months — and the auditor will sample evidence across that period. Trust Services Criterion CC7.2 ("the entity monitors system components and the operation of those components") means the audit log itself is one of the artifacts the auditor inspects. Losing the prior 12 months means failing the audit.

Many SOC 2 reports stretch to 18 or 24 months across consecutive review periods so an auditor can verify "you said you did this last year, here's the evidence." 18 months is the safe Velox default — it covers a full Type 2 cycle plus enough buffer that an auditor's mid-cycle resampling doesn't run off the end.

If the same Velox deployment is also serving a financial-services tenant whose own SOX program requires 7-year retention, the SOX window dominates — there's no point retaining for 18 months and then facing a 7-year subpoena with no evidence.

### GDPR / EU privacy

**Recommended: keep what's necessary to demonstrate accountability, with documented purpose. The audit log itself is personal data when it references actor identifiers.**

Reasoning: GDPR Article 5(1)(e) ("storage limitation") says personal data must be kept "for no longer than is necessary for the purposes." The purpose for an audit log is **accountability** — Article 5(2) requires the controller to be able to *demonstrate* compliance with the other principles. A short-retention audit log collapses that ability.

But the audit log is itself personal data: rows reference `actor_id` (often a real person's API key), `ip_address` (personal data per WP29 / Art 4), and may carry a customer's `display_name` in `resource_label` or `metadata`. So the retention window is two-sided:

- **Long enough** to demonstrate accountability for any reasonably anticipated dispute or regulator request — typically 1–3 years in the EU privacy literature.
- **Short enough** that you're not retaining personal data past the demonstrable need.

Velox's posture: 18 months in the live `audit_log` table satisfies the accountability test for typical SaaS operations; archived / glaciered exports are kept longer with clear retention policy in your data-mapping document. When a `gdpr.delete` action runs, the *customer*'s data is anonymised in the live tables, but the audit row recording that action is retained — that's evidence of compliance and is exempt under the storage-limitation principle when the purpose is "demonstrating Article 17 erasure was performed." Cross-link to the (Week-10) `docs/compliance/gdpr-data-export.md` once it lands for the operator-facing erasure flow.

### PCI-DSS

**Recommended: 1 year minimum, 3 months immediately accessible (per PCI-DSS v4 Requirement 10.5.1).**

Reasoning: Velox does not store cardholder data — Stripe holds the card; Velox holds tokens. The PCI scope of a Velox deployment is therefore limited. But audit rows recording API-key rotation (`rotate` on `api_key`), payment-method changes on a customer, and webhook-secret rotations touch the boundary of cardholder-data systems. PCI-DSS Requirement 10 ("Log and Monitor All Access to System Components and Cardholder Data") applies to this boundary.

The 1-year / 3-month rule from §10.5.1: at minimum 1 year of audit logs retained, with at least the most recent 3 months "immediately available for analysis." For Velox that maps cleanly onto: 18 months in `audit_log` (live, queryable), with the last 3 months cached on hot storage (the table itself, indexed and indexed). PCI-relevant rows specifically — `rotate`, payment-setup updates, API-key creates / deletes — should be archived to immutable storage (S3 Object Lock or equivalent) before the live table rolls off.

### HIPAA

**Recommended: 6 years if the tenant handles PHI; otherwise N/A.**

Reasoning: HIPAA §164.316(b)(2)(i) requires retention of policies, procedures, and documentation "for 6 years from the date of its creation or the date when it last was in effect, whichever is later." Audit logs of access to ePHI fall under §164.312(b) ("Audit Controls"). Velox itself is not a healthcare system, but a Velox tenant may serve a healthcare customer whose contract requires HIPAA-aligned retention.

The healthcare-tenant edge case: if any tenant on this Velox deployment has signed a BAA with their downstream customer that flows HIPAA obligations through, set the live `audit_log` retention to 6 years for *that tenant only* (per-tenant retention isn't yet a configurable knob — see "Configuration knobs" below — so until then, archive monthly to immutable storage and rely on the archive for the 6-year window).

### Financial / SOX

**Recommended: 7 years.**

Reasoning: SOX §802 requires retention of records "for a period of 5 years from the end of the fiscal period," extended in practice to 7 years by SEC guidance and most audit firms' standard programs. IRS publication 583 + state revenue codes typically also land on 7 years for invoice-level financial records. The corresponding audit-log entries — invoice `finalize`, `void`, `refund`, credit-note `issued` — must outlive the financial record they describe.

`audit_log` is NOT the financial ledger (that's `invoices`, `invoice_line_items`, `payments`, and `customer_credit_ledger`). But the audit row that says "user X voided invoice INV-0042 on 2026-04-18" is part of the integrity story for that financial record. Keep audit rows for financial actions for the same window as the underlying financial records: **7 years**, archived (Glacier / S3 Object Lock with 7-year retention), not necessarily live.

### Default Velox recommendation

| Tier | Live `audit_log` table | Archived (S3 / Glacier) |
|---|---|---|
| Default | **18 months** | Indefinite, with bucket-lifecycle expiration at **7 years** |
| SOC 2 Type 2 only | 18 months | 3 years (covers two consecutive Type 2 cycles + slack) |
| Financial / SOX | 18 months | 7 years (immutable; S3 Object Lock recommended) |
| HIPAA | 6 years live, or 18 months live + 6 years immutable archive | 6 years immutable |
| PCI-DSS boundary | 18 months | 1 year minimum, 7 years recommended |
| GDPR / EU | 18 months | "As long as necessary, no longer" — document the purpose |

The 18-month live default is the tradeoff between **query speed** (the live table stays small enough for sub-second queries on the indexed columns) and **compliance availability** (any auditor's "show me the last year" sample lands inside the live window). Anything older lives in archive and is restored when a specific question is asked, not browsed routinely.

## Operational pruning + archival

### Partitioning decision

`audit_log` is **not partitioned today**. Migration `0001_schema.up.sql` creates it as a plain table, and no follow-up migration has changed that.

Tradeoff:

- **Monthly partitions** (e.g. `audit_log_2026_04`, `audit_log_2026_05`) make pruning trivial: `DROP TABLE audit_log_<old>` is metadata-only, no row scan, no lock on the hot table. Drawback: every audit insert becomes a partition-routing decision, and the FORCE-RLS policies have to be re-applied per partition. The migration to convert an existing populated table to a partitioned one is non-trivial: it's a `CREATE TABLE … PARTITION BY` plus a backfill, and the constraint validation pass on an existing 5M-row table has the same lock problem migration 0015 solved for foreign keys. (See `docs/migration-safety-findings.md` for the measurement methodology and the NOT-VALID + VALIDATE pattern that 0015 used to dodge `AccessExclusiveLock`.)
- **No partitioning + monthly batched DELETE** is the simpler model for tenants under ~10M `audit_log` rows/year. A single Velox deployment with 100 active tenants writing 1 audit row/sec each generates ~3.15B rows/year — at that scale, partition. Below ~10M rows/year, the batched DELETE strategy keeps lock contention bounded with no schema complexity.

**Velox ships unpartitioned.** When a deployment crosses ~5M rows/month sustained (visible in `pg_class.reltuples` for `audit_log`), revisit and partition. Until then, use the prune pattern below.

### Pruning the live table — batched DELETE pattern

The audit log immutability trigger (`audit_log_immutable_trg`) blocks all DELETEs by default. Legitimate retention pruning therefore runs as a DB-admin operation: drop the trigger, prune in batches (so no statement holds locks for more than a few seconds at a time), recreate the trigger.

```sql
-- Run as the DB owner. The whole sequence is a DB-admin operation;
-- the operator's audit trail is in the Postgres statement log
-- (log_statement = 'ddl' or higher), not in audit_log itself.

BEGIN;
  -- Confirm exactly what you are about to delete BEFORE dropping the trigger.
  SELECT count(*) FROM audit_log
   WHERE created_at < now() - INTERVAL '18 months';
COMMIT;

-- 1. Drop the immutability trigger.
DROP TRIGGER IF EXISTS audit_log_immutable_trg ON audit_log;

-- 2. Batched delete loop. Each iteration deletes at most 10k rows so no
--    transaction holds row locks longer than a few hundred ms even on
--    a busy table. The CTE pattern is the pgsql idiom for "delete LIMIT N":
--    the bare DELETE statement does not accept a LIMIT clause, but
--    DELETE ... USING <subquery> does.
DO $$
DECLARE
    deleted INTEGER := 0;
    total   INTEGER := 0;
BEGIN
    LOOP
        WITH victims AS (
            SELECT id FROM audit_log
             WHERE created_at < now() - INTERVAL '18 months'
             ORDER BY created_at
             LIMIT 10000
             FOR UPDATE SKIP LOCKED
        )
        DELETE FROM audit_log
         WHERE id IN (SELECT id FROM victims);

        GET DIAGNOSTICS deleted = ROW_COUNT;
        total := total + deleted;
        EXIT WHEN deleted = 0;

        -- Yield between batches so concurrent INSERT traffic doesn't queue up.
        PERFORM pg_sleep(0.1);
    END LOOP;
    RAISE NOTICE 'Pruned % audit_log rows', total;
END$$;

-- 3. Recreate the immutability trigger. The function definition is
--    idempotent (CREATE OR REPLACE), so the original migration's body
--    is the source of truth — re-issuing it here costs nothing.
CREATE OR REPLACE FUNCTION audit_log_immutable()
RETURNS TRIGGER AS $$
BEGIN
    RAISE EXCEPTION
        'audit_log is append-only; % is not permitted', TG_OP
        USING HINT = 'To purge rows for retention, drop the trigger temporarily at the DB-admin level.';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_log_immutable_trg
BEFORE UPDATE OR DELETE ON audit_log
FOR EACH ROW EXECUTE FUNCTION audit_log_immutable();

-- 4. Reclaim the space. A normal autovacuum will catch up, but on a
--    multi-million-row prune you want the index bloat removed sooner.
VACUUM (VERBOSE, ANALYZE) audit_log;
```

The batch size (`LIMIT 10000`), per-batch `pg_sleep(0.1)`, and `FOR UPDATE SKIP LOCKED` mirror the same "don't lock the hot table" discipline that migration 0015 used (see [migration-safety-findings.md](../migration-safety-findings.md)) — concurrent INSERTs proceed unblocked because each prune transaction holds row-level locks on at most 10k rows for ~100ms before yielding.

### Cron / scheduled job

Velox does not ship its own cron for audit pruning. The deployment-side crontab is one line:

```cron
# Monthly audit-log prune, first Sunday at 04:00 UTC, 18-month retention.
# Logs to syslog tag velox-audit-prune for the deployment's log-aggregation pipeline.
0 4 1-7 * 0  PGPASSWORD="$VELOX_DB_PASSWORD" psql -h "$VELOX_DB_HOST" -U "$VELOX_DB_OWNER" -d velox -f /etc/velox/audit-prune.sql 2>&1 | logger -t velox-audit-prune
```

The `1-7 * 0` syntax (day-of-month 1-7 AND day-of-week Sunday) means "first Sunday of the month" — the standard cron idiom for monthly cadence on a fixed weekday.

Run this against the **primary**, not a replica — replicas can't write. Schedule it during a maintenance window so the brief absence of the immutability trigger doesn't surprise anyone reading audit_log statistics.

A tracked alternative is to add a Velox-internal scheduler that owns this job (similar to the `velox_scheduled_cleanup_rows_total` metric exposed for other retention jobs — see [`runbook.md`](./runbook.md#background-cleanup)). That work is gated on a tenant asking; until then, the cron above is the reference.

### Archival to S3

The pattern is `COPY` to a gzipped local file, upload to S3 with a date-prefixed key, and verify the upload before the prune deletes the rows. Run **before** the prune script above so that anything entering the archive bucket has already been written to S3 with checksum verification.

```bash
#!/bin/bash
# /etc/velox/audit-archive.sh — runs the day before the prune.
# Exports rows older than 18 months to a gzipped TSV uploaded to S3,
# then verifies the upload.
set -euo pipefail

CUTOFF="$(date -u -v-18m '+%Y-%m-%d')"   # macOS / BSD; on Linux: -d '18 months ago'
ARCHIVE_PREFIX="audit-log/${CUTOFF}/audit_log_pre_${CUTOFF}.tsv.gz"
ARCHIVE_LOCAL="/tmp/$(basename "${ARCHIVE_PREFIX}")"

# 1. Export. COPY to STDOUT keeps the export off-disk for the
#    intermediate rows; gzip on the fly minimises the tmpfile.
PGPASSWORD="${VELOX_DB_PASSWORD}" psql \
    -h "${VELOX_DB_HOST}" -U "${VELOX_DB_OWNER}" -d velox \
    -c "COPY (SELECT id, tenant_id, actor_type, actor_id, action,
                    resource_type, resource_id, resource_label,
                    metadata::text, ip_address, request_id, created_at
              FROM audit_log
              WHERE created_at < '${CUTOFF}'::timestamptz)
        TO STDOUT WITH (FORMAT csv, HEADER true, DELIMITER E'\t')" \
    | gzip -9 \
    > "${ARCHIVE_LOCAL}"

# 2. Upload with a content-MD5 so S3 verifies bytes-on-the-wire.
MD5=$(openssl dgst -md5 -binary "${ARCHIVE_LOCAL}" | base64)
aws s3 cp "${ARCHIVE_LOCAL}" "s3://${VELOX_AUDIT_ARCHIVE_BUCKET}/${ARCHIVE_PREFIX}" \
    --content-md5 "${MD5}" \
    --metadata "tenant=all,cutoff=${CUTOFF}"

# 3. Verify the object exists at the expected size.
LOCAL_SIZE=$(stat -f%z "${ARCHIVE_LOCAL}" 2>/dev/null || stat -c%s "${ARCHIVE_LOCAL}")
REMOTE_SIZE=$(aws s3api head-object \
    --bucket "${VELOX_AUDIT_ARCHIVE_BUCKET}" \
    --key "${ARCHIVE_PREFIX}" \
    --query 'ContentLength' --output text)
[ "${LOCAL_SIZE}" = "${REMOTE_SIZE}" ] || {
    echo "size mismatch local=${LOCAL_SIZE} remote=${REMOTE_SIZE}" >&2
    exit 1
}

rm -f "${ARCHIVE_LOCAL}"
echo "archived audit_log rows pre ${CUTOFF} to s3://${VELOX_AUDIT_ARCHIVE_BUCKET}/${ARCHIVE_PREFIX}"
```

S3 bucket lifecycle policy — apply once at bucket creation:

```json
{
  "Rules": [
    {
      "Id": "VeloxAuditArchiveTransitionToGlacier",
      "Status": "Enabled",
      "Filter": { "Prefix": "audit-log/" },
      "Transitions": [
        { "Days": 90, "StorageClass": "GLACIER_IR" },
        { "Days": 365, "StorageClass": "DEEP_ARCHIVE" }
      ],
      "Expiration": { "Days": 2555 }
    }
  ]
}
```

`2555` days ≈ 7 years, matching the conservative SOC 2 / SOX upper bound. For tenants with HIPAA flow-through (6 years from "last in effect"), bump to `2920` days. For PCI-only deployments where 1 year is the floor, the same lifecycle policy is fine — keeping 7 years of compressed TSV in Deep Archive at ~$1/TB/month is operationally cheaper than re-deciding the policy later.

For tenants who require **immutability of the archive** (SOX, regulated finance), enable [S3 Object Lock](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html) in compliance mode on the bucket with the same retention period. Object Lock prevents even the bucket owner from deleting an object before its retention expiry — the audit-archive equivalent of the `audit_log_immutable_trg` trigger, one storage layer up.

### Restore process

When an auditor or operator needs a window of archived rows back in a query-able form, the canonical path is to restore into a **separate** `audit_log_archive` table — never the live `audit_log`. The live table's primary-key collisions and immutability trigger make a re-import there mechanically painful; a side table avoids that and keeps the restore operationally reversible (just `DROP TABLE audit_log_archive` when the audit closes).

```sql
-- One-time setup the first time you do a restore. The schema mirrors
-- audit_log but has no triggers and no FK to tenants — restored rows
-- describe historical state and shouldn't fail to load if a tenant
-- has since been hard-deleted (which itself is gated by ON DELETE
-- RESTRICT on the live table; in practice tenants are never
-- hard-deleted, but archive restores must be robust to anything).
CREATE TABLE IF NOT EXISTS audit_log_archive (
    LIKE audit_log INCLUDING DEFAULTS,
    archived_from_key TEXT  -- which S3 key this row came from
);
CREATE INDEX IF NOT EXISTS idx_audit_log_archive_resource
    ON audit_log_archive (tenant_id, resource_type, resource_id);
CREATE INDEX IF NOT EXISTS idx_audit_log_archive_created
    ON audit_log_archive (tenant_id, created_at DESC);
```

Then for each restore window:

```bash
# 1. Pull the archive object from S3 (will rehydrate from Glacier if needed —
#    Glacier IR rehydrates in ~minutes; Deep Archive in 12-48h).
KEY="audit-log/2024-10-15/audit_log_pre_2024-10-15.tsv.gz"
aws s3 cp "s3://${VELOX_AUDIT_ARCHIVE_BUCKET}/${KEY}" /tmp/restore.tsv.gz
gunzip /tmp/restore.tsv.gz

# 2. Load. The COPY is into the audit_log_archive side table. Wrap in a
#    transaction so a partial load doesn't leave half-restored rows.
psql -d velox <<SQL
BEGIN;
  CREATE TEMP TABLE _stage (LIKE audit_log INCLUDING DEFAULTS);
  \copy _stage FROM '/tmp/restore.tsv' WITH (FORMAT csv, HEADER true, DELIMITER E'\t');
  INSERT INTO audit_log_archive
  SELECT *, '${KEY}' FROM _stage
  ON CONFLICT (id) DO NOTHING;  -- restores are idempotent at the row level
COMMIT;
SQL

rm -f /tmp/restore.tsv
```

Querying restored rows is the same as querying the live table, just substitute the table name:

```sql
SELECT * FROM audit_log_archive
WHERE tenant_id = $1
  AND resource_type = 'subscription' AND resource_id = $2
ORDER BY created_at DESC;
```

When the auditor finishes, drop the archive table — it's purely a query convenience. The S3 archive remains the system of record.

## Querying the audit log

The dashboard exposes `GET /v1/audit-log` and `GET /v1/audit-log/filters` — gated by `auth.PermAPIKeyRead`, served by `internal/audit/handler.go`. The frontend client is `web-v2/src/lib/api.ts::listAuditLog` / `getAuditFilters`. Both surface the canonical filter set: resource type, resource id, action, actor id, date range. The query implementation is `internal/audit/audit.go::Query` — note the `LEFT JOIN api_keys` to surface `actor_name` so the UI shows "Production" instead of `vlx_secret_live_…`.

For ad-hoc investigation outside the dashboard, the indexed columns are `(tenant_id, resource_type, resource_id)` and `(tenant_id, created_at DESC)`.

```sql
-- Every action by a specific API key in the last 30 days.
SELECT al.created_at, al.action, al.resource_type, al.resource_id,
       al.resource_label, al.metadata, k.name AS actor_name
FROM audit_log al
LEFT JOIN api_keys k ON k.id = al.actor_id AND k.tenant_id = al.tenant_id
WHERE al.tenant_id = 'tnt_…'
  AND al.actor_id  = 'vlx_secret_live_…'
  AND al.created_at >= now() - INTERVAL '30 days'
ORDER BY al.created_at DESC;

-- Every change to a specific subscription, full history.
SELECT created_at, actor_type, actor_id, action, metadata
FROM audit_log
WHERE tenant_id     = 'tnt_…'
  AND resource_type = 'subscription'
  AND resource_id   = 'sub_…'
ORDER BY created_at;

-- Every API-key rotation in the last 90 days (PCI-relevant).
SELECT created_at, tenant_id, actor_type, actor_id, resource_id, resource_label
FROM audit_log
WHERE action      = 'rotate'
  AND resource_type = 'api_key'
  AND created_at  >= now() - INTERVAL '90 days'
ORDER BY created_at DESC;

-- Every GDPR delete operation, ever.
SELECT created_at, tenant_id, resource_id, metadata
FROM audit_log
WHERE action = 'gdpr.delete'
ORDER BY created_at DESC;

-- Trace a customer-reported request id back to the audit row.
-- The customer hits a 503 audit_error and quotes the Velox-Request-Id
-- header; you join that to the chi-emitted request id stored on the row.
SELECT *
FROM audit_log
WHERE request_id = '01J9F9D…';
```

### A note on PII

Audit rows contain personal data:

- `actor_id` resolves through `api_keys.name` to a human-set label that may be a person's name.
- `ip_address` is personal data per WP29 / EDPB guidance.
- `metadata` and `resource_label` may carry customer display names, email addresses (if a handler chose to record them), invoice numbers tied to a person, or other PII.

**Never share raw audit log exports with anyone outside the operator team without redacting actor identifiers and per-row metadata.** When responding to a GDPR Article 15 (right of access) or Article 20 (portability) request, route through the `gdpr.go` exporter — it knows which fields are safe to share with the data subject. `audit_log` itself is not part of the user-facing GDPR export today; that's deliberate. Auditors who need the raw audit feed sign an NDA + data-processing agreement that covers the personal data the log contains.

## Configuration knobs

The grep is short — there are no env vars or config flags that affect audit logging today. The only knob is `tenant_settings.audit_fail_closed`, set per-tenant via the existing settings API, which decides whether an audit-write failure becomes a 503 or a quiet metric increment (default: fail-open).

Future work, not implemented yet:

- **`VELOX_AUDIT_RETENTION_DAYS`** — global default for the live-table retention window. Today the cron's `INTERVAL '18 months'` is the source of truth; a future scheduler-internal job would read this env var instead. Document this when the in-binary scheduler lands.
- **`VELOX_AUDIT_ARCHIVE_BUCKET`** — implicit in the archive script above; the binary itself does not consume this. If a built-in archive worker ships, it will read this env var and the matching IAM credentials.
- **`tenant_settings.audit_retention_days`** — per-tenant override. Required for the HIPAA flow-through case described above (a single Velox deployment serving one healthcare-flow tenant alongside default tenants). Until this exists, a HIPAA-flow tenant should be operated as a separate Velox deployment, or all tenants' retention bumped to 6 years (with the cost implications).

Track these against the Week-10 follow-on ticket once the surrounding compliance docs (encryption-at-rest verification, SOC2 control mapping, GDPR data export) land — all four are interconnected and the in-binary surface is best designed once.
