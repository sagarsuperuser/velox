# Migration safety findings — populated-DB pass

> **Scope.** A one-shot hardening measurement run before Phase 3 (production
> rollout, Week 9+). All 57 migrations were applied to a populated scratch
> database; per-migration wall-clock duration and `AccessExclusiveLock`
> hold-times on hot tables were recorded for both the up and the down
> directions. Findings + recommended fixes for the migrations that would
> hurt in production are below.
>
> **Authorship.** Generated by `cmd/velox-migrate-safety` (Go program) +
> `scripts/migration-safety-test.sh` (driver). Re-run any time. Not wired
> into CI — a medium-scale pass takes ~6 minutes, too slow for every push.
>
> **No new migrations were written.** This document specifies the fixes;
> a follow-up PR applies them.

## TL;DR

| Migration | Up duration (5M events) | Hot-table lock | Risk |
|-----------|-------------------------|----------------|------|
| **0054 multi_dim_meters** | 53.5s | `usage_events` 53.5s | **CRITICAL** — full table rewrite of largest table (BIGINT→NUMERIC) + non-concurrent GIN index |
| **0020 test_mode** | 14.5s | `invoices` 14.4s | **HIGH** — adds livemode column + drops/re-adds 13 UNIQUE constraints across 32 tables in one step |
| **0015 fk_explicit_restrict** | ~~8.9s~~ → 0.26s (small) | ~~`audit_log` 8.8s~~ → **0ms** | ✅ **FIXED** — rewritten to NOT VALID + VALIDATE; see "Already fixed" below |
| **0017 usage_events_origin** | 1.3s | (no lock observed >50ms) | MEDIUM — fast-default but CHECK validates every row |
| **0014 tax_rate_bp_bigint** | 0.75s | (no lock observed >50ms) | LOW — comment claims metadata-only, actual is small but non-zero |

The down direction is symmetrically painful: rolling back 0054 took 49.8s and
held an `AccessExclusiveLock` on `meter_pricing_rules` for the entire window.

**Net effect on production:** running these migrations against a real database
with ≥5M usage_events would freeze writes to `usage_events` for ~minute (0054)
and to `invoices` for ~15s (0020). 0015 was the third blocker (9s lock on
`audit_log`) and has now been fixed (NOT VALID + VALIDATE rewrite, lock drops
to 0ms — see "Already fixed" below). Before Phase 3, **0054 and 0020 must
still be reworked into multi-step non-blocking variants** (recipes below).
The remaining 53 migrations are safe.

## Methodology

### Workflow

1. Drop & recreate scratch DB (`velox_migrate_safety` by default).
2. Apply migration `0001_schema.up.sql` only — base schema.
3. Bulk-seed populated data via multi-row INSERTs, with `app.bypass_rls=on`:
   - 20 tenants × 1000 customers per tenant = 20k customers
   - 500 subs per tenant = 10k subs (capped to ≤ customers/tenant to satisfy
     migration 0013's UNIQUE on (tenant, customer, plan))
   - 500 usage events per sub = 5M `usage_events` rows
   - 10 invoices per sub × 10k subs = 100k `invoices` rows
   - 5000 audit rows per tenant = 100k `audit_log` rows
4. Walk forward 0002..0057, applying each migration individually. For every
   step, time wall-clock and concurrently sample `pg_locks` every 50ms for
   `AccessExclusiveLock` on the curated hot-table list.
5. Walk backward 0057..0002, identical instrumentation.
6. Emit CSV report.

The seed strategy deliberately keeps tables that do not yet exist at version
0001 empty — they fill up only once the migration that creates them runs. The
hot-table risk story is unchanged: 0001's tables (`usage_events`, `invoices`,
`subscriptions`, `customers`, `audit_log`) are fully populated when every
later migration runs against them.

### Hot-table lock instrumentation

A side goroutine on its own connection runs every 50ms:

```sql
SELECT DISTINCT c.relname
  FROM pg_locks l
  JOIN pg_class c ON c.oid = l.relation
 WHERE l.mode = 'AccessExclusiveLock'
   AND l.granted = TRUE
   AND c.relname IN ('usage_events', 'invoices', 'invoice_line_items',
                     'subscriptions', 'customers', 'audit_log',
                     'customer_credit_ledger', 'webhook_outbox',
                     'email_outbox', 'meter_pricing_rules');
```

For each table, the first sighting marks `firstSeen[t]`. When the lock
disappears, we log `now - firstSeen[t]` as the contiguous lock duration. The
maximum over the migration's lifetime is recorded.

**Caveat.** The 50ms sample interval will under-count locks shorter than ~50ms
and slightly under-count near the boundaries. For the migrations that matter
(>1s lock) this is well within tolerance.

### Scale presets

| Preset | Events | Invoices | Audit | Wall-clock |
|--------|--------|----------|-------|------------|
| tiny   | 24     | 12       | 20    | ~3s        |
| small  | 500k   | 125k     | 50k   | ~50s       |
| medium | 5M     | 100k     | 100k  | ~6min      |
| large  | 20M    | 200k     | 400k  | ~25min     |

### Hardware

- macOS, Postgres 16.13 in Docker (default `shared_buffers`, no tuning)
- M-series ARM CPU
- Default Docker shm size — sufficient for `ANALYZE`; insufficient for full
  `VACUUM ANALYZE` at 5M rows (the harness runs `ANALYZE` only).

A production cluster on dedicated SSDs with tuned `shared_buffers`,
`maintenance_work_mem`, and `effective_io_concurrency` will be 2-5× faster.
The relative ranking between migrations is what the report is for; absolute
wall-clock numbers should be considered upper bounds for laptop-class
hardware.

## Per-migration findings

### Already fixed

#### 0015 — `fk_explicit_restrict` — ✅ FIXED (2026-04-26)

Originally drops + re-adds every FK in the schema as `ON DELETE RESTRICT`.
60 FKs total across ~30 tables. The atomic `DROP CONSTRAINT … ADD CONSTRAINT
FOREIGN KEY …` shape validated every existing row under `AccessExclusiveLock`
— measured at 8.9s up / 6.8s down on `audit_log` at the medium scale. The
migration has been rewritten in-place to use the standard NOT VALID +
VALIDATE two-step on every FK so validation moves to
`ShareUpdateExclusiveLock` (PG 9.4+) — concurrent INSERT/UPDATE/DELETE
proceed unblocked. No runner-flag changes needed; the whole sequence still
runs inside golang-migrate's outer transaction.

- **Before (medium, 5M events).** Up 8.9s, AccessExclusive on `audit_log`
  for 8.8s. Down 6.8s, AccessExclusive on `audit_log` for 6.7s.
- **After (small, 500k events — same harness, same hot-table sampler).**
  Up 0.26s, **no `AccessExclusiveLock` observed**. Down 0.24s, **no
  `AccessExclusiveLock` observed**. The lock-time floor is the harness's
  50ms sample interval; the actual hold time is below that.
- **Process note.** The original migration was correct for an empty
  database (atomic drop+add inside one statement). NOT VALID + VALIDATE
  is the standard production rewrite once tables are populated. Phase 3
  prep applied in PR
  [#43](https://github.com/sagarsuperuser/velox/pull/43).

### Top risks

#### 0054 — `multi_dim_meters` — CRITICAL

Two operations on `usage_events`, the largest table in the schema:

```sql
ALTER TABLE usage_events
    ALTER COLUMN quantity TYPE NUMERIC(38, 12) USING quantity::numeric;

CREATE INDEX idx_usage_events_properties_gin
    ON usage_events USING GIN (properties);
```

- **Up duration.** 53.5s at 5M rows. AccessExclusive on `usage_events`
  for the entire 53.5s.
- **Down duration.** 49.9s at 5M rows. AccessExclusive on
  `meter_pricing_rules` for 49.8s while it drops + the type reverts.
- **Why it hurts.** `ALTER COLUMN ... TYPE NUMERIC USING ::numeric` is a
  full table rewrite — Postgres must rewrite every row, every index, every
  toast pointer. Concurrent writes are blocked. The migration's own comment
  (line 30) acknowledges this:
  > _quantity column rename: BIGINT -> NUMERIC is a metadata-only change
  > when no rows exist (pre-launch), and a rewrite when rows exist._
  This was a conscious "we're pre-launch" decision. After Phase 3 we are
  not pre-launch and the migration must be reworked.
- **Recommended fix (production-safe variant).** Three migrations:
  1. **0054a** — `ADD COLUMN quantity_decimal NUMERIC(38, 12)` (no rewrite).
  2. **0054b** — Backfill loop in application code: batches of 10k rows
     `UPDATE usage_events SET quantity_decimal = quantity::numeric WHERE
     quantity_decimal IS NULL LIMIT 10000` until zero rows match.
  3. **0054c** — Once backfill is complete and verified:
     `ALTER TABLE usage_events RENAME COLUMN quantity TO quantity_legacy;
      ALTER TABLE usage_events RENAME COLUMN quantity_decimal TO quantity;
      ALTER TABLE usage_events ALTER COLUMN quantity SET NOT NULL;`
     (the `SET NOT NULL` is a constraint validation, not a rewrite, and is
     fast on a column where every row has a value already).
- **GIN index fix.** `CREATE INDEX CONCURRENTLY idx_usage_events_properties_gin`
  — must run outside a transaction, so this requires either a "no-tx"
  migration flag (golang-migrate supports this via a header comment) or
  applying it with a small custom runner. CONCURRENTLY does not block writes.

#### 0020 — `test_mode` — HIGH

```sql
ALTER TABLE api_keys ADD COLUMN livemode BOOLEAN NOT NULL DEFAULT true;
-- ... 32 more tables, including `invoices` and `subscriptions`
ALTER TABLE customers DROP CONSTRAINT customers_tenant_id_external_id_key;
ALTER TABLE customers ADD UNIQUE (tenant_id, livemode, external_id);
-- ... 13 UNIQUE swaps total
```

- **Up duration.** 14.5s. AccessExclusive on `invoices` for 14.4s.
- **Down duration.** 8.4s. AccessExclusive on `customers` for 8.3s.
- **Why it hurts.** The DEFAULT is a constant (`true`), so on Postgres 11+
  the `ADD COLUMN` itself is fast-default (metadata-only). The slow part
  is the 13 `DROP CONSTRAINT ... ADD UNIQUE` swaps: each `ADD UNIQUE`
  builds a new index, which is a non-concurrent index build that holds
  `AccessExclusiveLock` for the duration. Stacked across 13 tables, this
  adds up.
- **Recommended fix.** Split into two migrations:
  1. **0020a** — All `ADD COLUMN livemode BOOLEAN NOT NULL DEFAULT true`
     (fast-default metadata-only on every table). RLS-policy rewrites
     also go here (DDL on `pg_policy`, no table lock).
  2. **0020b** — Per UNIQUE swap, a separate migration:
     ```
     CREATE UNIQUE INDEX CONCURRENTLY new_uniq_idx
         ON customers (tenant_id, livemode, external_id);
     ALTER TABLE customers
         DROP CONSTRAINT customers_tenant_id_external_id_key,
         ADD CONSTRAINT customers_tenant_livemode_external_id_key
             UNIQUE USING INDEX new_uniq_idx;
     ```
     `CREATE INDEX CONCURRENTLY` does not block writes. The DROP+ADD
     CONSTRAINT step is fast-metadata once the index exists.

  The 13 swaps don't all need separate migrations — group them in one
  migration that uses `CREATE UNIQUE INDEX CONCURRENTLY` + ATTACH. CI
  remains green; production rollout is an order of magnitude safer.

### Medium risks

#### 0017 — `usage_events_origin` — MEDIUM

```sql
ALTER TABLE usage_events
    ADD COLUMN origin TEXT NOT NULL DEFAULT 'api'
        CHECK (origin IN ('api', 'backfill'));
```

- **Up duration.** 1.3s at 5M rows.
- **No `AccessExclusiveLock` >50ms observed.** Fast-default applies.
- **Why slower than expected.** The CHECK constraint requires Postgres to
  validate that *every existing row* satisfies the predicate. Although the
  default `'api'` matches the CHECK, PG still scans the table to verify.
  With 5M rows + a wide row size (JSONB properties, timestamps, FKs),
  this is ~1.3s of pure read-only work — but no lock release/acquisition
  cycles, so writes weren't blocked beyond the metadata-update window.
- **Recommended fix (optional).** Two migrations:
  ```sql
  -- 0017a — DEFAULT but no CHECK.
  ALTER TABLE usage_events
      ADD COLUMN origin TEXT NOT NULL DEFAULT 'api';

  -- 0017b — NOT VALID then VALIDATE.
  ALTER TABLE usage_events
      ADD CONSTRAINT usage_events_origin_chk
          CHECK (origin IN ('api', 'backfill')) NOT VALID;
  ALTER TABLE usage_events
      VALIDATE CONSTRAINT usage_events_origin_chk;
  ```
- **Priority.** Lower than the top three. Even at 50M rows the impact is
  ~10s of CPU work but no `AccessExclusiveLock` is held for that whole
  duration — only the metadata update at the start.

#### 0014 — `tax_rate_bp_bigint` — LOW

```sql
ALTER TABLE tenant_settings ALTER COLUMN tax_rate_bp TYPE BIGINT;
ALTER TABLE invoices ALTER COLUMN tax_rate_bp TYPE BIGINT;
ALTER TABLE invoice_line_items ALTER COLUMN tax_rate_bp TYPE BIGINT;
```

- **Up duration.** 0.75s. **Down duration.** 0.8s with 745ms lock on
  `invoices`.
- **Comment claim.** "ALTER … TYPE BIGINT is a cheap in-place metadata
  change on Postgres 12+ when the source type is INT (no rewrite required)."
- **Reality.** True for the type cast itself, but PG still acquires
  `AccessExclusiveLock` to swap the column header, and the down direction
  (BIGINT → INT) **does** rewrite if any value exceeds INT4 range.
- **Production recommendation.** Keep the up as-is (low risk at 100k
  invoice scale; even at 10M it's a sub-second hit). Add a comment noting
  the down direction will rewrite if values exceed INT4 max — operators
  rolling back must verify `MAX(tax_rate_bp) <= 2^31-1` first.

### No-issue migrations

Every other migration completed in under 100ms at 5M-row scale with no
hot-table `AccessExclusiveLock` exceeding 50ms. Most of these create new
tables (which never block existing writes) or add columns with constant
DEFAULTs (PG fast-default). Notable categories:

- 0002 (seed feature flags), 0021 (livemode trigger), 0023 (email_bidx),
  0028 (tax_home_country), 0031 (tax_provider), etc.: all add a column
  with a constant DEFAULT or seed a small lookup row.
- 0026 (subscription_items), 0029 (subscription_item_changes),
  0034 (embedded_auth), 0038 (tax_calculations), 0043 (coupons_v2),
  0046 (customer_coupon_assignment), 0055 (recipe_instances),
  0056 (subscription_billing_thresholds), 0057 (billing_alerts):
  all create new tables with their own RLS + indexes; no impact on the
  populated hot tables.

### Down-direction-only risks

Beyond the symmetric down-pass costs of 0054 / 0020 / 0015 / 0014 noted
above, the rollback pass uncovered no migrations that **silently drop
data**. Every column-removal `down.sql` either drops a column where the
data is recoverable from another column (e.g. 0014 INT4 narrowing —
operator-verifiable beforehand) or drops a column that was added in the
matching `up.sql` (no original data to lose).

The roundtrip test (`internal/platform/migrate/roundtrip_test.go`) already
catches the case where a `.down.sql` doesn't fully undo its `.up.sql`. No
new failures here.

## RLS / advisory-lock checks

- **RLS policies are unchanged on rollback.** Migration 0020 rewrites the
  `tenant_isolation` policy on 32 tables to add the `livemode` predicate.
  Down-migration 0020 restores the original predicate. The CSV shows down
  0020 took 8.4s with `AccessExclusive` on `customers` for 8.3s — i.e.
  the same UNIQUE-rebuild cost mirrored in reverse. **Recommendation:** if
  rolling back 0020 in production is ever needed, follow the same
  multi-step pattern (drop new index concurrently first, then DROP+ADD
  CONSTRAINT, then drop the column).
- **Advisory lock for migration ordering.** golang-migrate takes
  `pg_advisory_lock` keyed on db+schema before applying. Since this
  harness runs serially (1 migration at a time), no advisory-lock
  contention was observed. Production rolling deploys with multiple
  replicas booting concurrently will serialize on this lock — they will
  briefly all block until the lock holder finishes. If a 0054-class
  migration is in flight, a deploy-time timeout might fire on the waiting
  replicas. Solution: run migrations as a dedicated deploy-step (k8s Job),
  not on app boot — `migrate.go` already documents this in its `Up`
  comment.

## Re-running this pass

```bash
docker compose up -d postgres

# Quick smoke (validates harness):
scripts/migration-safety-test.sh tiny

# Default — sufficient for catching most regressions:
scripts/migration-safety-test.sh small

# Recommended pre-Phase-3 pass:
scripts/migration-safety-test.sh medium

# Override individual knobs:
SCALE="tenants=10,customers_per_tenant=100,subs_per_tenant=50,events_per_sub=200,invoices_per_sub=5,audit_per_tenant=500" \
  scripts/migration-safety-test.sh small
```

The CSV is written to `/tmp/velox-migration-safety-${PRESET}.csv`. Diff
against the prior snapshot to spot regressions.

The harness program lives at `cmd/velox-migrate-safety/`. Build tag: none —
it operates only on a scratch DB it owns; safe to invoke from any
worktree.

Total wall-clock for the **medium** pass on a laptop: ~6 minutes (3:20 of
which is seeding 5M rows). The migration-time portion is ~2:30 forward +
~1:00 backward. If we wanted to run this nightly via cron in Phase 3, the
medium preset is fast enough; the **large** preset (20M events, ~25min)
is overkill except as a one-off pre-launch sanity check.
